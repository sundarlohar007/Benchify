    **Part A — Python Frida injection tooling:**

    **frida/gadget_injector.py** — Pure-Python APK modification (no apktool needed):
    - Function `inject_frida_gadget(apk_path, gadget_so_path, output_path, arch=None)`:
      1. Open APK as ZIP file (APK = ZIP format). Validate it's a valid ZIP.
      2. Detect target architecture by reading `lib/` directory contents. If `lib/arm64-v8a/` exists → arch is arm64. If `lib/armeabi-v7a/` exists → arch is arm. If both, prefer arm64.
      3. Copy `frida-gadget-<arch>.so` into `lib/<abi>/libgadget.so` inside the ZIP.
      4. If the APK already has `lib/<abi>/libgadget.so`, overwrite it (build system may have included it).
      5. Write modified ZIP to `output_path`.
      6. Do NOT sign the APK — Frida gadget injection intentionally leaves original signature intact (per D-07: only Smali path re-signs).
    - Function `get_arch_from_apk(apk_path)` → returns "arm64", "arm", or "x86_64" based on lib directory contents.
    - Uses Python `zipfile` module exclusively — no shell commands.

    **frida/gadget_config_template.json** — JSON configuration for frida-gadget:

      "interaction": {
        "type": "listen",
        "address": "127.0.0.1:27042",
        "on_load": "resume"

    **Part A — WebView JS Memory Collection (V25-08, D-15):**

    **WebViewBridge.java** — Java/Kotlin class providing `addJavascriptInterface` bridge:
    ```java
    package dev.benchify;

    import android.webkit.JavascriptInterface;
    import android.webkit.WebView;

    public class WebViewBridge {
        private static final String JS_INTERFACE_NAME = "__benchify";
        private boolean installed = false;

        // JNI callback to Rust
        public static native void nativeReportJsHeap(int usedHeapKb);

        /**
         * Install the JS bridge into the given WebView.
         * Call this from the app's onCreate after initializing SDK.
         * Safe to call multiple times — only installs once per WebView instance.
         */
        public void install(WebView webView) {
            if (installed) return;
            webView.getSettings().setJavaScriptEnabled(true);
            webView.addJavascriptInterface(new JsBridge(), JS_INTERFACE_NAME);
            installed = true;
        }

        /**
         * JavaScript interface exposed to web content.
         * Called via: window.__benchify.reportMemory(usedHeapKB);
         */
        public class JsBridge {
            @JavascriptInterface
            public void reportMemory(int usedHeapKb) {
                nativeReportJsHeap(usedHeapKb);
            }
        }

        // Periodic collection via evaluateJavascript
        private static final String JS_PROBE =
            "(function(){" +
            "  if(window.performance && window.performance.memory) {" +
            "    var used = window.performance.memory.usedJSHeapSize;" +
            "    window.__benchify.reportMemory(Math.floor(used/1024));" +
            "  }" +
            "})()";

        public void probeJsMemory(WebView webView) {
            webView.post(() -> {
                webView.evaluateJavascript(JS_PROBE, null);
            });
        }
    }
    ```

    **webview_js.rs** — Rust collector for JS heap data:
    - Module `metrics::webview_js` with a static `AtomicI32` storing latest `used_heap_kb`.
    - JNI function `Java_dev_benchify_WebViewBridge_nativeReportJsHeap` updates the atomic.
    - On each 1Hz tick, `get_webview_metrics()` reads the atomic and returns `Option<(i32)>` for `memory_webview_kb`.
    - Include the value in the unified MetricSample JSON as `memory_webview_kb`.

    **lib.rs (modify)** — Add JNI export:
    ```rust
    #[no_mangle]
    pub extern "system" fn Java_dev_benchify_WebViewBridge_nativeReportJsHeap(
        _env: JNIEnv, _class: JClass, used_heap_kb: jint
    ) {
        metrics::webview_js::report_js_heap(used_heap_kb);
    }
    ```

    **Part B — Per-Process Network Stats (V25-09, D-16):**

    **net_per_process.rs** — Per-process network parser:
    - Module `metrics::net_per_process` reading `/proc/<pid>/net/dev`.
    - On `init(pid: u32)`, store the PID.
    - On `collect(pid: u32)`, read `/proc/<pid>/net/dev` (or fall back to `/proc/self/net/dev`).
    - Parse format:
      ```
      Inter-|   Receive                                                |  Transmit
       face |bytes    packets errs drop fifo frame compressed multicast|bytes    packets errs drop fifo colls carrier compressed
      wlan0: 12345678 1000 0 0 0 0 0 0  87654321 800 0 0 0 0 0 0
      rmnet0: 123456 50 0 0 0 0 0 0  654321 30 0 0 0 0 0 0
      ```
    - Track cumulative deltas since last collection (per-second rates).
    - Classify interfaces: `wlan*` → wifi, `rmnet*` → cellular, everything else → other.
    - Return `NetPerProcessResult` with fields: `net_tx_bytes`, `net_rx_bytes`, `net_wifi_tx_bytes`, `net_wifi_rx_bytes`, `net_cellular_tx_bytes`, `net_cellular_rx_bytes`, `net_other_tx_bytes`, `net_other_rx_bytes`.
    - **IMPORTANT per D-16:** This is process-level — NOT socket-level interception. The data comes from `/proc/pid/net/dev` which provides per-interface counters for the process's network namespace. This is less granular than socket interception but requires no root.

    **mod.rs (modify)** — Register new modules:
    ```rust
    pub mod webview_js;
    pub mod net_per_process;
    ```

    **lib.rs (modify)** — In the metric collection tick, call `net_per_process::collect(pid)` and include results in the unified MetricSample JSON.

    **IMPORTANT — Per-process /proc access:** Android apps running in their own process can read `/proc/self/net/dev` but may NOT be able to read `/proc/<other_pid>/net/dev` unless running as root. Since the SDK runs inside the target app's process, it reads its own network stats via `/proc/self/net/dev`. This provides the app's own network usage (not device-wide). This is the correct behavior per D-16.

    **IMPORTANT — WebView detection:** The `WebViewBridge.install()` method requires the target app to call it when setting up WebViews. The SDK cannot auto-detect all WebViews. Document this in the SDK README: "For WebView memory monitoring, call `WebViewBridge.install(webView)` after creating each WebView."

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Frida gadget → App process | frida-gadget.so loads into target app — has full process access |
| JavaScript in WebView → Java bridge | `addJavascriptInterface` exposes native methods to web content |
| /proc filesystem reads | Process-level network stats from /proc — same process, no privilege escalation |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-13 | Elevation of Privilege | frida/gadget_injector.py | accept | Frida gadget injection is intentionally less secure than re-signing (no signature verification after injection). User accepts this tradeoff per D-09. |
| T-04-14 | Information Disclosure | WebViewBridge.java | mitigate | `addJavascriptInterface` only exposes `reportMemory(int)` — single method, validated input (positive integer only, max 1GB). No file access, no shell commands exposed to JS. |
| T-04-15 | Tampering | benchify_frida_agent.js | accept | Frida gadget config uses `listen` mode — any local process can connect to frida-server. This is inherent to Frida's architecture. Developer tool running on dev device. |
| T-04-16 | Information Disclosure | net_per_process.rs | accept | Reads only the current process's /proc/self/net/dev — no cross-process data access. Same information the app itself has access to. |
| T-04-17 | Denial of Service | webview_js.rs (AtomicI32) | mitigate | JNI callback runs on binder thread — use AtomicI32 (lock-free) for stats. No allocations or blocking operations in the JNI bridge path. |
</threat_model>
