**Thread-level CPU breakdown** (`server/src/routes/sessions.rs` — add endpoint, V35-08):
- `GET /api/v1/sessions/{session_id}/cpu-threads` — Returns per-thread CPU breakdown
- This is a server-side aggregation endpoint. The actual thread-level CPU data comes from the PC profiling agent (Phase 5 `pb-pcprobe`). For Android/iOS targets, thread CPU data is not available.
- Response format:
  ```json
  {
    "available": true|false,
    "requires_root": true,
    "note": "Thread-level CPU breakdown requires root/administrator access on the target device",
    "threads": [
      {
        "thread_name": "RenderThread",
        "tid": 12345,
        "cpu_percent_avg": 12.5,
        "cpu_percent_peak": 45.2,
        "user_time_ms": 45000,
        "kernel_time_ms": 2300,
        "sample_count": 120
      }
    ],
    "total_threads": 42,
    "collection_duration_ms": 60000
  }
  ```
- If the session has thread CPU data in its metric_samples JSONB, extract and aggregate it. Otherwise, return `available: false` with note about root requirement.
- The endpoint checks if `session_stats->'pc_metrics'->'thread_cpu'` exists in the JSONB column. If present, it extracts and aggregates thread samples. If absent, returns the "not available" response.
- Thread CPU data format (from Phase 5 PC metrics): Each MetricSample may include `pc_metrics.thread_cpu: [{ tid, thread_name, cpu_percent, user_time_ms, kernel_time_ms }]`.
- Aggregation: group by (tid, thread_name), compute avg/peak cpu_percent, sum user_time_ms and kernel_time_ms, count samples.

| Boundary | Description |
|----------|-------------|
| Web dashboard → Server API | Admin-level API calls (user management, SSO config, audit purge) cross trust boundary — role enforced server-side |
| Server → Jira API | Server initiates outbound HTTPS call to Jira Cloud with API token — token stored in server config |
| Kubernetes ingress → Server | External traffic enters through ingress controller — TLS should terminate at ingress |
| Browser → File download | Audit export downloads CSV/JSON via browser — Content-Disposition prevents inline rendering |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-16 | Spoofing | web: /admin/users role dropdown | mitigate | Role change is validated server-side via admin middleware + RBAC check. Client-side role dropdown is UX only — server authorizes the PUT request. |
| T-06-17 | Information Disclosure | web: /admin/audit AuditLogTable | mitigate | Audit page requires admin or auditor role (checked at server). Client-side route guard via ProtectedRoute + role check prevents rendering for unauthorized users. |
| T-06-18 | Information Disclosure | routes/jira.rs | mitigate | Jira API token stored in server config (not exposed to client). Client sends session_id to server; server fetches session data and constructs Jira payload server-side. Jira token never leaves server. |
| T-06-19 | Denial of Service | web: /admin/audit export | mitigate | Audit export queries are paginated on the backend. CSV/JSON streaming limits memory usage. Export endpoint rate-limited by same mechanism as other API endpoints. |
| T-06-20 | Elevation of Privilege | web: Sidebar admin links | accept | Sidebar conditionally renders admin links based on user.role from server. This is a client-side UX convenience — actual authorization is server-side via RBAC middleware. Malicious user could manually navigate to /admin/users URL but server returns 403. |
